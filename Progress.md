# Распределение задач (Модульный монолит)

Архитектура: **Модульный монолит** (Один Spring Boot 3 сервис, Java 25, PostgreSQL).
Команда: 5 человек. Распределение задач выполнено так, чтобы каждый участник занимался преимущественно разработкой на Java, отвечая за свой изолированный модуль. Модули взаимодействуют друг с другом через вызовы Java-интерфейсов внутри одного приложения.

## 1. Впишите себя — Chat Module (Core RAG Pipeline)  & Core Infrastructure
**Ответственность:** Реализация главного функционала чата и взаимодействия с LLM (Генерация ответов).
**Задачи на Java:**
- Разработка REST API для чата (`ChatController`, веб-сокеты или SSE для стриминга ответов).
- Интеграция с LangChain4j для вызова LLM (OpenAI / Anthropic).
- Сборка промптов (Prompt Engineering) на основе контекста.
- Оркестрация RAG: получение вопроса → запрос векторного поиска у `Indexing Module` → формирование контекста → запрос к LLM → возврат ответа с атрибуцией (ссылками на источники).
- Управление историей диалогов (сохранение контекста бесед в БД).
- Инфраструктура: написание `docker-compose.yml` (PostgreSQL + pgvector) и скриптов для локального запуска проекта всей командой.

## 2. Впишите себя — Indexing Module (Парсинг, Чанкинг и Вектора)
**Ответственность:** Извлечение текста из файлов, векторизация и векторный поиск в БД.
**Задачи на Java:**
- Реализация парсеров (`DocumentParser`) для извлечения текста из PDF (Apache PDFBox), DOCX (Apache POI) и TXT.
- Разбиение текста на смысловые фрагменты (TextChunker, настройка overlap и размера чанка).
- Генерация embeddings для чанков через LangChain4j/LLM API.
- Настройка асинхронных фоновых задач (`@Async`, `@Scheduled`) для обработки очередей файлов на индексацию без блокировки основного потока.
- Реализация `VectorSearchService` — выполнение SQL-запросов к PostgreSQL (`pgvector`) для поиска ближайших соседей (similarity search) с учетом фильтров доступа.

## 3. Впишите себя — Storage Module (Облачные хранилища)
**Ответственность:** Интеграция с внешними провайдерами дисков и скачивание файлов.
**Задачи на Java:**
- Реализация API клиента для **Google Drive API** (обход дерева папок, скачивание файлов).
- Реализация API клиента для **Яндекс.Диск API**.
- Безопасное хранение и автоматическое обновление (refresh) OAuth-токенов для подключенных дисков.
- Отслеживание изменений на дисках (webhook/polling), чтобы триггерить переиндексацию только для новых или измененных файлов.
- Передача скачанных файловых потоков (InputStreams) в `Indexing Module`.

## 4. Впишите себя — Auth Module & Core Infrastructure
**Ответственность:** Безопасность, аутентификация пользователей и фундамент проекта.
**Задачи на Java:**
- Настройка и конфигурация **Spring Security**.
- Реализация процесса авторизации (Login) через Google OAuth 2.0 для пользователей.
- Генерация, валидация и управление сессиями через **JWT-токены**.
- Настройка общих DTO, глобального перехватчика ошибок (`@ControllerAdvice`), логирования.

## 5. Впишите себя — Admin Module & Data Access (Группы и Доступы)
**Ответственность:** Бизнес-логика управления правами, группами и структурой базы данных.
**Задачи на Java:**
- Разработка REST API для панели администратора (управление пользователями и дисками).
- Реализация бизнес-логики **Групп**: создание групп, добавление участников, привязка подключенных облачных дисков к группам (для создания общих баз знаний).
- Проектирование схемы реляционной БД: написание миграций **Liquibase** или **Flyway** (таблицы users, groups, memberships, drives, documents).
- API для мониторинга: отдача на Frontend статусов индексации документов (проценты загрузки, ошибки).
- Гарантия изоляции данных: предоставление механизмов проверки прав доступа для `Chat Module` и `Indexing Module` (пользователь может искать только по своим или групповым файлам).

## Технологии проекта
* **Backend:** Java 25, Spring Boot 3
* **AI & RAG Framework:** LangChain4j (интеграция с LLM, генерация эмбеддингов, оркестрация RAG)
* **База данных и вектора:** PostgreSQL 16 + расширение `pgvector`
* **Парсинг документов:** Apache PDFBox (для PDF), Apache POI (для DOCX). Выбор на усмотрение того, кто будет делать
* **Аутентификация:** Spring Security, OAuth 2.0 (Google), JWT
* **Среда запуска и развертывания:** Docker, Docker Compose
* **Управление миграциями БД:** Liquibase или Flyway
* **Frontend (отдельно):** React навайбкодим

## О чем нужно договориться сразу (Внутренние контракты)
Поскольку мы делаем Модульный монолит, все модули живут в одной кодовой базе, но чтобы не мешать друг другу, нужно до начала написания бизнес-логики согласовать следующее:

1. **Единые модели данных (DTO) на стыке модулей:**
   * В каком виде передаются файлы между модулями? (Например, договориться, что `Storage Module` отдает `InputStream` вместе с метаданными файла в `Indexing Module`, а не сохраняет промежуточные файлы на жесткий диск).
   * Структура ответа от `Indexing Module` к `Chat Module`. Как именно возвращаются найденные релевантные фрагменты текста (чанки) и ссылки на источники для LLM?
2. **Изоляция данных и передача контекста пользователя:**
   * Как мы передаем ID пользователя (`user_id`) или группы (`group_id`) сквозь все слои приложения? (Самый частый паттерн в Spring — доставать его из `SecurityContextHolder`).
   * **Критически важно:** `VectorSearchService` должен обязательно принимать `user_id`/`group_id` от `Chat Module` при поиске, чтобы один пользователь не смог через чат вытащить данные из чужих документов.
3. **Единая обработка ошибок:**
   * Необходимо договориться о едином формате JSON-ответа с ошибками для Frontend.
   * Написать общие классы исключений (например, `EntityNotFoundException`, `AccessDeniedException`), которые будет перехватывать глобальный `@ControllerAdvice`
4. **Управление схемой БД:**
   * Так как БД одна на всех, схему таблиц нужно обсуждать и проектировать вместе. Никто не должен втихую менять общие таблицы (например, `users` или `documents`) через автогенерацию Hibernate. Все изменения должны идти строго через SQL-миграции (Liquibase/Flyway)
5. **Структура пакетов в проекте:**
   * Договоритесь о жесткой структуре папок (например, `com.project.chat`, `com.project.storage`, `com.project.common`), чтобы модули были физически разделены внутри монолита, и никто не делал циклических зависимостей между пакетами.
